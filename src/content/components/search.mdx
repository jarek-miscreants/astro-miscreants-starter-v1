Lightweight client-side search. Drop the input above any list of items, point it at them with a CSS selector, and the script hides anything that doesn't match the query. Matching is fuzzy (subsequence) with bonuses for prefix matches and word-boundary hits, so `"btn"` finds **B**u**t**to**n**, `"drp"` finds **D**ropdown, etc.

It's wired into the live [Components index](/components/) at the top of the page — try it there to see fuzzy scoring against this exact list.

## Wiring it to a list

The component is decoupled from the list it filters — you mark items with attributes and pass selectors. Three pieces:

1. **Items** carry `data-search-item` and (optionally) `data-search-text`. If `data-search-text` is omitted, the item's `textContent` is used. Custom text lets you index hidden facets (category, tags, alternate names).
2. **Groups** (optional) wrap items and carry the group selector (e.g. `data-search-group`). When every item inside a group is hidden, the group is hidden too — so empty section headers don't linger.
3. **Empty state** (optional) is any element marked with the empty selector — hidden by default, revealed when zero items match.

```astro
---
import Search from "@components/Search.astro";
---

<Search
  targetSelector="[data-search-item]"
  groupSelector="[data-search-group]"
  emptySelector="[data-search-empty]"
  placeholder="Search…"
/>

<section data-search-group>
  <h2>Primitives</h2>
  <ul>
    <li data-search-item data-search-text="Button primary action CTA">…</li>
    <li data-search-item data-search-text="Tag chip pill label">…</li>
  </ul>
</section>

<p data-search-empty hidden>No results.</p>
```

## Wiring to a content collection

Items rendered from a content collection just need their text concatenated into `data-search-text`. The full setup used by the live [Components index](/components/):

```astro
---
import { getCollection } from "astro:content";
import Search from "@components/Search.astro";

const all = await getCollection("components");
const groups = ["primitive", "block", "pattern"].map((cat) => ({
  category: cat,
  items: all.filter((c) => c.data.category === cat),
}));
---

<Search
  targetSelector="[data-search-item]"
  groupSelector="[data-search-group]"
  emptySelector="[data-search-empty]"
  placeholder="Search components…"
/>

{groups.map((group) => (
  <section data-search-group>
    <h2>{group.category}</h2>
    <ul>
      {group.items.map((item) => (
        <li
          data-search-item
          data-search-text={`${item.data.title} ${item.data.description} ${item.data.category}`}
        >
          <a href={`/components/${item.id}/`}>
            <h3>{item.data.title}</h3>
            <p>{item.data.description}</p>
          </a>
        </li>
      ))}
    </ul>
  </section>
))}

<div data-search-empty hidden>No components match.</div>
```

Same shape works for any collection — blog tags, FAQ entries, changelog items.

## Scoring

The fuzzy scorer returns a value in `[0..1]`:

| Score   | Meaning                                                            |
|---------|--------------------------------------------------------------------|
| `1.0`   | Query matches at the start of the target string.                   |
| `0.9`   | Query matches a word boundary (after space, `-`, `_`, `/`).        |
| `0.7`   | Query matches mid-word as a contiguous substring.                  |
| `0–0.6` | Subsequence match — query characters in order, with gaps allowed.  |
| `0`     | At least one query character couldn't be matched.                  |

Items pass when their score is greater than `minScore`. With the default `minScore=0.01` everything that's at least a subsequence match shows up; raise it to `0.7` if you only want substring matches.

## Events

Each filter pass dispatches a `search:filter` event on the wrapper:

```js
document.addEventListener("search:filter", (e) => {
  console.log(e.detail);
  // { query: "btn", visible: 1, total: 47 }
});
```

Use it to update a result counter, log analytics, or persist the query to the URL.

## Accessibility

- Real `<input type="search">` — gets `Cmd/Ctrl+F`-style browser affordances and supports the native Escape-clear behaviour.
- `<label>` is visually hidden by default but always present — screen readers announce the input's purpose.
- Polite `aria-live` status region announces `"N results for 'query'"` after each filter pass, so screen-reader users know the list shrank.
- Clear button is a real `<button>` with `aria-label="Clear search"`. Hidden when the input is empty, so the tab order isn't cluttered.
- The browser's built-in WebKit clear UI is suppressed in CSS so we don't double up.
- Items are hidden via the `hidden` attribute, which removes them from both the visual flow and the accessibility tree.

## Notes

- Filtering happens entirely client-side, so the full list must be rendered server-side first. For very large lists (thousands of items) consider a server-rendered search route instead — fuzzy matching across thousands of nodes per keystroke will lag.
- `data-search-text` is plain text — no HTML. Concatenate whatever facets you want indexed.
- Hidden items remain in the DOM, so deep-linking and "view source" still work as expected.
- The scorer is deliberately small (≈40 lines) and dependency-free. If you outgrow it (typo tolerance, weighted fields, multi-word AND), swap to [Fuse.js](https://fusejs.io/) or [MiniSearch](https://lucaong.github.io/minisearch/) and feed results back via `hidden` toggles using the same selector contract.
