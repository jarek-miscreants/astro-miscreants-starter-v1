Lightweight tooltip built on the native [Popover API](https://developer.mozilla.org/en-US/docs/Web/API/Popover_API). Uses `popover="manual"` so its visibility is controlled by hover/focus, not by click — that's the right model for a hint, not an action menu. `aria-describedby` is wired automatically from the trigger to the tooltip.

## Slots

| Slot         | Purpose                                                              |
|--------------|----------------------------------------------------------------------|
| (default)    | The trigger element. The first focusable descendant gets `aria-describedby`. |
| `trigger`    | Alternative explicit trigger slot — used when `content` slot is also provided. |
| `content`    | Rich content for the tooltip body. Overrides `text`.                 |

## Accessibility

- Tooltip element gets `role="tooltip"`.
- The trigger gets `aria-describedby` pointing at the tooltip — screen readers announce the description after the trigger label.
- `popover="manual"` means clicks elsewhere do **not** dismiss the tooltip — that's the right behaviour for a description, not a menu.
- Opens on `focus` (keyboard users) as well as `pointerenter`. Closes on `blur` and `pointerleave`.
- Escape closes the tooltip without moving focus.
- `prefers-reduced-motion: reduce` removes the fade/slide transition.
- Tooltips have `pointer-events: none` so they never intercept clicks on the underlying UI.

## Dependencies

None. Native Popover API (Baseline 2024).

## Notes

- The trigger should be a real focusable element (`<button>`, `<a>`, `<input>`, etc.). Tooltips on non-focusable text are a screen-reader trap; if you need one, add `tabindex="0"` to the trigger.
- Tooltips close on scroll and resize — re-hovering re-opens them at the new position.
- Multiple tooltips on a page are independent and can be triggered from different elements with no shared state.
