Compact author block — avatar circle (image or initials) plus an optional name and position label. Image is optional; if omitted or it fails to load, initials derived from the name are shown instead. An optional status dot can be pinned to the corner of the avatar.

## Accessibility

- The avatar figure is `role="img"` and gets an `aria-label` of `name` when `hideLabel` is set, so screen readers always announce who it represents.
- When the visible name is shown, the figure's accessible name is left empty (the name appears as text right next to it) — this avoids duplicate announcement.
- Status dots get `role="status"` and an `aria-label` like `"Online"` so screen readers describe presence.
- Initials are marked `aria-hidden` — they're a visual fallback, not part of the accessible name.
- The image uses `loading="lazy"` and `decoding="async"` to keep avatars off the critical path.

## Wiring to MD / MDX content

There are three common patterns for driving Avatars from authored content.

### 1. Inline inside MDX

MDX files can import the component and use it directly — same as inside this doc:

```mdx
---
title: "Why we ship every Friday"
date: 2026-04-12
---

import Avatar from "@components/Avatar.astro";

<Avatar
  name="Jane Cooper"
  src="/img/team/jane.jpg"
  position="Principal Designer"
/>

The team has been shipping every Friday for the past two years…
```

This works for the body of any MDX page (blog posts, case studies, changelog entries) because MDX evaluates `import` and JSX. Plain `.md` files cannot do this — Astro's Markdown renderer doesn't run JSX.

### 2. Frontmatter-driven (recommended for blog posts / case studies)

Type the author shape on the collection so every entry can pass it to `<Avatar />` from a layout. In `src/content.config.ts`:

```ts
const blog = defineCollection({
  loader: glob({ pattern: "**/*.{md,mdx}", base: "./src/content/blog" }),
  schema: z.object({
    title: z.string(),
    date: z.coerce.date(),
    author: z.object({
      name: z.string(),
      position: z.string().optional(),
      avatar: z.string().optional(),
    }),
  }),
});
```

A blog post's frontmatter:

```md
---
title: "Why we ship every Friday"
date: 2026-04-12
author:
  name: Jane Cooper
  position: Principal Designer
  avatar: /img/team/jane.jpg
---

Body copy here…
```

Render it from the page route (e.g. `src/pages/blog/[...slug].astro`):

```astro
---
import Avatar from "@components/Avatar.astro";
import { getEntry, render } from "astro:content";

const entry = await getEntry("blog", Astro.params.slug);
const { Content } = await render(entry);
const { author } = entry.data;
---

<header>
  <h1>{entry.data.title}</h1>
  <Avatar
    name={author.name}
    src={author.avatar}
    position={author.position}
    size="sm"
  />
</header>
<Content />
```

This works for both `.md` and `.mdx` posts because the layout — not the body — renders the avatar.

### 3. Shared author records (referenced from posts)

If the same person writes many posts, define an `authors` collection and reference it by id from each post:

```ts
const authors = defineCollection({
  loader: glob({ pattern: "**/*.md", base: "./src/content/authors" }),
  schema: z.object({
    name: z.string(),
    position: z.string().optional(),
    avatar: z.string().optional(),
  }),
});

const blog = defineCollection({
  loader: glob({ pattern: "**/*.{md,mdx}", base: "./src/content/blog" }),
  schema: z.object({
    title: z.string(),
    date: z.coerce.date(),
    author: reference("authors"),
  }),
});
```

`src/content/authors/jane-cooper.md`:

```md
---
name: Jane Cooper
position: Principal Designer
avatar: /img/team/jane.jpg
---
```

A post references `author: jane-cooper`, and the page resolves it:

```astro
---
import Avatar from "@components/Avatar.astro";
import { getEntry } from "astro:content";

const post = await getEntry("blog", Astro.params.slug);
const author = await getEntry(post.data.author);
---

<Avatar
  name={author.data.name}
  src={author.data.avatar}
  position={author.data.position}
/>
```

Editing one author file updates the avatar everywhere.

## Notes

- Initials are derived as: first letter of the first word + first letter of the last word, uppercased. Single-word names use the first two letters. Empty / whitespace-only names render `?`.
- The image is layered over the initials; if it fails to load, a small client script hides the image so the initials show through. Cached errors before the listener attaches are also handled.
- The status dot uses `box-shadow` to draw a ring matching `--color-canvas`, so it stands out against any avatar.
- For groups of overlapping avatars (a stack), wrap several `Avatar`s in a flex container with negative left margins on all but the first.
